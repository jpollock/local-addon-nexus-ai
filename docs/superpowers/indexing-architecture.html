Gaps, Issues & Problem Areas

Critical blocking first-run

🔴
Halted sites silently fail to index
INDEX_SITE requires the MySQL socket file which only exists when a site is running. Called on a halted site: extracts 0 posts, marks state as "indexed" with documentCount=0, no error shown. User sees 0 results with no explanation. INDEX_ALL_AUTO is the correct channel but was not used in the Discover tab until recently.
🔴
WPE sites return zero results from Discover search
SEARCH and SEARCH_KEYWORD only query LanceDB, which contains locally-indexed content only. WPE installs are never included regardless of sync status. A user with 20 WPE sites and 0 local sites gets zero results. The graph DB has a content table but it's not wired to any search handler.
🔴
No real-time progress during indexing
ContentPipeline has an onStatusChange callback that fires with stage, progress %, and message (e.g. "Embedding 40 of 200..."). This callback is never pushed to the renderer. DiscoverTab polls on a 10-second interval. Users see a static "waiting" grid for minutes with no indication progress is happening.

Major impacts reliability

🟡
ONNX model failure silently disables all indexing
If EmbeddingService.initialize() fails, readyPromise rejects. The lifecycle hook catches it, logs an error, and skips indexing for every site start until addon restart. No UI feedback. Users see sites as "waiting" indefinitely with no actionable message.
🟡
MySQL race condition on site start
siteStarted fires when nginx/PHP are ready, but MySQL initialization can take an additional 2-10 seconds. MySQLExtractor.connect() times out. The index completes with 0 documents — no retry until next site start. No user notification.
🟡
Three overlapping persistence stores — no unified "site state"
IndexRegistry (content index state), SiteMetadataCache (WP runtime state), and the graph DB sites table all hold overlapping data. They can drift. DiscoverTab must reconcile indexEntries from IndexRegistry with sites from Local's API — two separate IPC calls that can race.

Minor tech debt

⚪
INDEX_ALL_AUTO starts all sites and leaves them running
When triggered from Discover, users see all their sites suddenly start. A Discover-specific variant that stops sites after indexing would be better UX — especially for machines with limited RAM.
⚪
Docs and code have drifted
docs/architecture/digital-twin.md (606 lines) describes a planned architecture that doesn't match the current Discover tab integration or the WPE search gap. Plans predate INDEX_ALL_AUTO being the correct indexing path.

Rationalized Architecture — Principles

1
One unified "IndexedSite" concept
A single known state per site regardless of type: unindexed, indexing, metadata-only, or fully-indexed. DiscoverTab reads from one source — not three diverging stores.
2
Progressive search capability — not binary
Keyword search works as soon as any content is in the graph DB. Semantic search activates when vectors are built. WPE sites contribute keyword results even without full vector indexing.
3
Real-time progress, not polling
Wire ContentPipeline.onStatusChange → IPC push event → DiscoverTab listener. Updates every second, not every 10 seconds.
4
Explicit dependency declarations, loud failures
Each indexer declares preconditions. The system fails loudly with actionable UI messages — not silently. "ONNX model not ready" and "MySQL not available, will retry" rather than invisible skips.
5
Testable at every layer
Each indexer independently unit-testable with mocks. Integration tests use WPE_FIXTURES (jppwpeplugin, jppwpeplugistg). No test requires all 14 sites running simultaneously.

// ContentPipeline already fires this — needs wiring to renderer:
contentPipeline.onStatusChange = (siteId, status) => {
  mainWindow.webContents.send(IPC_CHANNELS.INDEX_PROGRESS, {
    siteId,
    state:         status.state,     // 'indexing' | 'indexed' | 'error'
    progress:      status.progress,   // 0-100
    message:       status.message,    // 'Extracting...', 'Embedding 40 of 200...'

// DiscoverTab.componentDidMount adds a listener:
ipc.on(IPC_CHANNELS.INDEX_PROGRESS, (_, data) => {
  this.setState(prev => ({
    indexEntries: prev.indexEntries.map(e =>
      e.siteId === data.siteId ? { ...e, ...data } : e
    )
  }));
});

interface SiteIndex {
  // ── Identity ──────────────────────────────────────────────────────
  siteId:             string;
  siteName:           string;
  source:             'local' | 'wpe';
  domain:             string;
  environment:        'production' | 'staging' | 'development' | null;
  remoteInstallId:    string | null;  // WPE install UUID
  accountId:          string | null;  // WPE account UUID

  // ── WordPress Runtime ─────────────────────────────────────────────
  wpVersion:          string | null;
  phpVersion:         string | null;
  siteUrl:            string | null;
  adminEmail:         string | null;
  activeTheme:        string | null;

  // ── Inventory ─────────────────────────────────────────────────────
  plugins:            Plugin[];
  themes:             Theme[];
  postCount:          number | null;
  postCountByType:    Record<string, number> | null;
  lastPostAt:         number | null;

  // ── Search Index ──────────────────────────────────────────────────
  searchCapability:   'none' | 'keyword' | 'semantic';
  indexState:         'unknown' | 'indexing' | 'indexed' | 'stale' | 'error';
  indexProgress:      number | null;   // 0-100 during indexing
  indexMessage:       string | null;   // "Embedding 40 of 200..."
  documentCount:      number;
  chunkCount:         number;
  lastIndexed:        number | null;
  indexErrors:        string[];

  // ── Freshness ─────────────────────────────────────────────────────
  lastUpdated:        number;
  updateSource:       'lifecycle' | 'manual' | 'setup-ai' | 'wpe-sync' | 'capi';
  isStale:            boolean;         // derived: lastUpdated > 24h
  wpeSyncedAt:        number | null;
  capiSyncedAt:       number | null;

  // ── Derived Capability Flags ──────────────────────────────────────
  // Computed from above fields — not stored, recalculated on read
  canKeywordSearch:   boolean;  // documentCount > 0 && ftsIndexExists
  canSemanticSearch:  boolean;  // searchCapability === 'semantic'
  canFleetAudit:      boolean;  // plugins.length > 0
  canVersionDrift:    boolean;  // wpVersion !== null
  canDbScan:          boolean;  // source === 'local' (site must be running)
  canWpCli:           boolean;  // local running OR wpe non-prod SSH
  canAiFeatures:      boolean;  // indexState === 'indexed' && aiProvider set
}
