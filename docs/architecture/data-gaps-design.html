1. Design Principles

Local-first, WPE opt-in The majority of users are Local-only developers. The product must be fast, automatic, and friction-free for them. WPE capabilities are additive — never assumed, never default-enabled.
WPE users control their SSH surface SSH connections to production environments are meaningful operations. Deep syncing (L2/L3 for WPE) must be explicitly enabled. CAPI calls are cheap and always-on; SSH is expensive and opt-in.
Data shown must be data trusted If a panel could be showing stale data, say so and tell the user what to do. A confident "0/19 Configured — click Refresh to fix this" is better than an unexplained 0.
One operation, full result When the user triggers any sync or index operation, it should be as complete as possible. L2 + L3 for WPE in one SSH session; L1 + L2 on local site-start. Users should not need to chain operations manually.

2. User Segments & Default Behaviour

~80%

Local-only developers

Use Local to build and test WordPress sites locally. No WPE account or connection. May have 1–20 sites.

What they need

  • L1 auto on startup (filesystem scan) — done, always runs
  • L2 auto on site-start (WP-CLI) — done, siteStarted hook
  • L3 on-demand or scheduled — opt-in checkbox
  • No WPE panels cluttering their view

Correct defaults

  • Content index interval: OFF — let them opt in
  • No WPE sections shown until WPE is connected
~20%

WPE customers

Connected to WP Engine. May have hundreds of installs. Sensitive about unexpected SSH connections to production.

What they need

  • CAPI L1 auto (install list, domain, php_version) — done, always runs
  • SSH L2 explicit opt-in — correct, default OFF
  • SSH L3 explicit opt-in, piggybacked on L2 — piggyback done, scheduler opt-in
  • Clear explanation of what each sync costs (SSH per site)

Correct defaults

  • Metadata sync (SSH): OFF — opt-in with explanation ✓
  • Content index for WPE: OFF / manual ✓
  • CAPI sync: always on — cheap, no SSH ✓
⚠ Key tension: WPE users want control, but the current experience gives them nothing by default — no wp_version, no plugins, no content — and doesn't explain why. The fix is not to enable more by default but to guide them toward opt-in with clear affordances.

High Phase 1

G8 — WPE Onboarding: Progressive disclosure for new WPE connections

~2 days

When a user first connects WPE (or sees 284 remote sites with all-zero L2/L3 data), they need a clear, honest explanation of what they have and what they can get — without automatically touching their production sites.

Trigger

Show on Dashboard when: remoteSites.total > 0 AND wpeConfigured === 0 AND onboarding.wpeDismissed !== true.

┌─ WP Engine Connected — 284 installs ──────────────────────────────────────────────────────┐ │ │ │ ✓ CAPI sync active — install list, domain, and PHP version are current (runs hourly). │ │ │ │ ⚡ Want plugin lists, WP version, and searchable content? │ │ That requires SSH access to each install. It's explicit and opt-in: │ │ │ │ [ Enable Metadata sync ] Get wp_version, plugins, users via SSH. Runs on your schedule. │ │ [ Index Content for WPE ] Make WPE sites searchable. Also syncs metadata. One-time or │ │ scheduled. │ │ │ │ Both are opt-in. Nothing runs until you say so. [ Dismiss ] │ └─────────────────────────────────────────────────────────────────────────────────────────────┘

Clicking Enable Metadata sync takes them directly to Settings → WPE Installs with the toggle highlighted. Clicking Index Content for WPE navigates to Operations and starts the bulk index.

Medium Phase 1

G9 + G10 — Freshness model & wp_version gap

~0.5 days

G9 — Freshness: The Site Status "0 fresh, 284 stale" label is misleading because CAPI sync (which runs hourly and populates php_version, domain, etc.) doesn't update last_sync_at. "Stale" currently means "no SSH sync" not "no recent data."

❌ Now

last_sync_at only updated by SSH sync.
CAPI runs hourly but sites show "stale".
284 stale even if CAPI just ran.
Hardcoded 8h threshold.

✅ Proposed

Introduce two freshness signals:
capi_synced_at — updated on every CAPI sync
ssh_synced_at — updated on SSH sync

"CAPI current" = capi_synced_at within 2h
"SSH current" = ssh_synced_at within configured interval
Show separately in Site Status.

G10 — wp_version: CAPI listInstalls doesn't include wp_version (only individual GET). Rather than making 284 individual CAPI calls on startup, wp_version should come exclusively from SSH L2 sync. The CAPI individual GET is not worth the API cost. Document this clearly in the UI: "WP version available after Metadata sync."

Updated Site Status display

Critical Phase 2

G6 — Shared Reactive Store (MobX)

~1 week
Context: Local already uses MobX 5 + mobx-react 6. These packages are in the runtime bundle. Our addon can use them without adding a dependency — just import from the host's module resolution.

Currently every component manages its own IPC fetch cycle. Data Completeness, Site Status, and Fleet Intelligence all poll independently on 30-second timers. When a sync completes, panels update at different moments, showing inconsistent states.

Proposed: NexusStore

A single MobX observable store in the renderer, populated by push events from the main process. All components become observers — they re-render automatically when the store changes, with no polling.

❌ Now (per-component polling)

FleetCompletenessWidget
  → IPC GET_FLEET_COMPLETENESS every 30s
  → renders with its own stale snapshot

SystemTab
  → IPC SYSTEM_WPE_STATUS every 30s
  → renders with DIFFERENT stale snapshot

NexusOverview
  → IPC GET_DASHBOARD_STATS on mount
  → renders with YET ANOTHER snapshot

Result: panels diverge after sync events

✅ Proposed (shared store)

Main process pushes events:
  IPC: NEXUS_STATE_UPDATED

NexusStore (MobX)
  @observable fleetCompleteness
  @observable wpeStatus
  @observable indexEntries
  @observable settings

All components are @observer
  → re-render automatically
  → always same snapshot
  → zero polling timers

Store structure

// src/renderer/store/NexusStore.ts import { observable, action, makeObservable } from 'mobx'; class NexusStore { @observable fleetCompleteness: FleetCompleteness | null = null; @observable wpeStatus: WpeStatus | null = null; @observable indexEntries: IndexEntry[] = []; @observable settings: NexusSettings = DEFAULT_SETTINGS; @observable syncProgress: SyncProgress | null = null; @action updateFromMain(event: NexusStateEvent) { // Called from ipcRenderer.on('nexus:state-update', ...) if (event.fleetCompleteness) this.fleetCompleteness = event.fleetCompleteness; if (event.wpeStatus) this.wpeStatus = event.wpeStatus; if (event.indexEntries) this.indexEntries = event.indexEntries; if (event.settings) this.settings = event.settings; } } export const nexusStore = new NexusStore();

Migration path (non-breaking)

  1. Create NexusStore with all current data shapes
  2. Add NEXUS_STATE_UPDATE push from main process after every sync/index event
  3. Convert components one at a time: replace IPC call + local state with store access
  4. Remove polling timers from converted components
  5. Keep old IPC handlers alive during transition — remove when all consumers migrated
Medium Phase 2

G11 — WPE L3 scheduled content indexing

~0.5 days

WPE content indexing is currently manual-only. Since L3 now piggybacks L2, a scheduled WPE content index interval covers both content freshness and metadata freshness in a single SSH session per site.

Add to Settings → WPE Installs: Content index for WPE checkbox + interval (default OFF). When enabled, indexAllWpeContent() runs on the configured interval and also refreshes L2 metadata.

Default must be OFF. SSH to production installs on a schedule requires explicit consent.
Medium Phase 3

G14 — LanceDB durability

~1 week (research-heavy)

LanceDB uses its own WAL mechanism (Lance format). On abnormal exit, the WAL may not be committed. Currently we have no recovery strategy — if vectors are lost, the user must re-index.

Short-term mitigation

  • Detect vector loss on startup: compare IndexRegistry document counts against actual LanceDB table row counts. If mismatch, mark affected entries as stale automatically.
  • Surface a "Some indexes appear corrupted — re-index?" banner rather than silently showing wrong data.

Long-term

  • Investigate LanceDB's optimize() / compaction API for explicit commit control
  • Consider checkpointing strategy: call optimize/compact before shutdown

5. Implementation Phases

Phase 1 — Clarity & Quick Wins

~1 week

No architectural change — improve what users see and understand with existing data.

  • G7 Data Completeness contextual action text
  • G8 WPE onboarding banner (first connection)
  • G9 Split CAPI/SSH freshness signals
  • G10 Document wp_version gap in UI ("available after Metadata sync")
  • G13 Settings: cost/consequence copy for WPE sync options
Phase 2 — Architecture

~2 weeks

Structural fix for data consistency. Highest-leverage investment.

  • G6 MobX NexusStore — migrate components incrementally
  • G6 Main process: push NEXUS_STATE_UPDATE events after every sync/index operation
  • G11 WPE L3 scheduled content index setting + scheduler
Phase 3 — Resilience & Polish

~2 weeks

Close remaining durability gaps. Lower urgency.

  • G14 LanceDB startup integrity check + stale detection
  • G12 Local L2: scheduled refresh for long-halted sites
  • Remove polling timers from all MobX-migrated components
  • LanceDB compaction/optimize before shutdown (research spike)
