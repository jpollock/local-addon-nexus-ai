nexusSitesStart/Stop for nonexistent sites hung for 120s

 Resolvers used wpeCached?.environment ?? 'production' — ignored the @staging in the target string. If install not in cache, always defaulted to production → blocked. Changed to parsed.environment ?? wpeCached?.environment ?? 'production' in all affected resolvers. ✓ Fixed · Playwright 13/13 pass
nexusWpPluginList missing access control

WPE sync progress — concurrent indexing

nexusStore re-hydration on renderer reconnect
 If the renderer reloads (e.g. addon disabled/re-enabled), the store starts empty. There's no mechanism to re-push current state from main → renderer on reconnect. Medium — affects users who disable/re-enable the addon. Store shows stale/empty data until next poll. ✗ Unproven

 4 tests skip because WPE OAuth is Keychain-encrypted. Need a CI-friendly auth mechanism (e.g., pre-shared test account with non-Keychain token storage, or Playwright auth state file). Medium — missing automated proof for the most important user-visible fix (B2) ⚠ Partial

WPE content indexing
SSH WP-CLI: posts/pages/CPTs → LanceDB
 Pulls posts via SSH WP-CLI and builds vector index for semantic search. This is what the user sees as "content pulling". Runs on schedule if enabled. Preferences → wpeContentIndexAutoEnabled (default: off) · wpeContentIndexIntervalHours
CAPI sync
REST API: account/install list
 Always runs on startup (fast, no SSH). Pulls install names, environments, domains from WPE API. No user toggle — this is read-only metadata. Always on · Cannot be disabled · <5s typically
If WPE content indexing is running unexpectedly, check Preferences → Settings → WPE Content Index Auto-run. The wpeContentIndexAutoEnabled setting defaults to false but may have been set to true in a previous session.
