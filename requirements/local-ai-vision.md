# Local as an Intelligent AI Host for WordPress Development

## Tagline

**Current:** "An effortless way to develop WordPress sites locally"
**New:** "WordPress and AI development, effortlessly local."

---

## The Core Vision

Local's original insight: abstract away painful WordPress infrastructure (nginx, MySQL, PHP, SSL, hosts file) behind something that just *works*.

The next chapter: **AI infrastructure is currently as painful to set up locally as WordPress was in 2012.** Local becomes the tool that makes AI infrastructure invisible too — running on your machine, across your entire fleet of WordPress sites.

**The positioning shift:**
> Local made WordPress easy to work with on one's computer.
> Local will now make WordPress *and AI* easy to work with on one's computer.

---

## The Mental Model: Local as Intelligent Multi-Site AI Host

Not just a dev tool. A host — running on your machine — that is intelligent about your entire WordPress practice. The AI infrastructure layer that sits beneath your sites, your tools, and your interfaces.

Key properties:
- **Fleet-aware**, not just site-aware
- **Interface-agnostic** — the host doesn't care if you're talking to it through Local UI, Claude, Claude Code, ChatGPT, or WP Admin
- **Local-first by default, cloud-connected by choice** — data and models stay on your machine unless you choose otherwise
- **Complementary to WordPress 7** — WP7 handles AI for a site's visitors and editors; Local handles AI for the developer and builder

---

## The AI Infrastructure Stack

Organized as layers, bottom up:

### Infrastructure
- **Vector DB** — local semantic search over your WordPress content, ACF fields, and site data; lightweight, runs on any machine. Embeddings generated by a bundled CPU-based model (~22MB quantized, ~90MB in memory, no GPU required) for baseline capability, with higher-quality embeddings available via cloud gateway or local LLM when configured. Per-site index footprint is ~12MB per 500 documents
- **Local LLM** — run models on-device for capable hardware; optional, not assumed

### Connectivity
- **MCP Server** — Local exposes your entire fleet as a well-designed MCP surface; tools and resources that any MCP-aware client (Claude, Cursor, etc.) can connect to natively
- **AI Gateway** — unified routing across local models, WP Engine cloud, and third-party APIs; one interface regardless of where inference actually happens. Note: this is the most architecturally complex component in the stack — it must handle model routing and fallback logic, credential management across multiple providers, cost tracking and usage visibility, and a consistent API surface regardless of backend. It should be scoped and sequenced carefully rather than delivered as a single monolithic service

### Capabilities
- **Skills / Tools** — reusable AI capabilities scoped to WordPress tasks
- **Agents** — autonomous workflows that operate on your sites
- **Content Pipelines** — structured workflows for AI-assisted content at scale

### Intelligence
- **Context Management** — what does the AI know about your site, your client, your project; maintained and current
- **Observability** — what are AI features actually doing; visibility into activity across the fleet
- **Evals / Testing** — regression testing for AI-powered sites; did that AI change actually work

## Tiered Capability Model

Local serves a wide range of users — from beginners on modest hardware to developers on powerful machines. The AI infrastructure adapts to what's available rather than assuming a high-capability environment.

**Tier 1 — Runs everywhere, no external dependencies**
Vector DB and MCP server. Lightweight, no GPU required, works on any machine Local supports. This is the foundation every user gets. Embeddings are generated locally using a bundled CPU-based embedding model (`all-MiniLM-L6-v2`, 22MB on disk, ~90MB in memory). Validated performance: 5ms per document, 184 docs/sec — a 500-post site indexes in under 3 seconds with negligible event loop impact (4ms P99). Semantic search works out of the box without any API calls or cloud dependency. Content is extracted via direct MySQL query when a site's database is running, supplemented by file-based scanning for structural context (themes, plugins, configuration). No worker threads needed — embedding generation is fast enough to stay on the main thread without affecting Local's UI responsiveness.

**Tier 2 — WP Engine Cloud AI Gateway**
LLM access, embeddings, and model management through an existing WP Engine account. No separate AI vendor relationship, no API key management, no model selection decisions. This is the "effortlessly" tier for the majority of users — the same gateway works in local development and in production, ensuring consistent behavior across environments.

**Tier 3 — Local LLM (capable hardware)**
On-device inference via Ollama for developers who want fully local, air-gapped, or maximum performance. The addon detects Ollama via a single HTTP request to `localhost:11434/api/tags` — if reachable, it lists installed models and recommends appropriate ones based on available RAM. On a 16 GB Apple Silicon machine, models up to 12B parameters (e.g., gemma3:12b) run well; 70B+ models require 64 GB+. If Ollama is not found, the addon shows "Install Ollama for local AI features" with a link — it does not bundle, auto-install, or manage the Ollama process. This is deliberate: many developers already run Ollama as a system service, and spawning a second instance fails on port conflict. Ollama's value in this architecture is specifically for LLM chat and completion — Tier 1's bundled ONNX embeddings are 5.4x faster than Ollama's embedding models and require no external dependency.

**Tier 4 — Bring your own API keys**
For developers with existing OpenAI, Anthropic, or other vendor relationships who want to route through those accounts via the Gateway layer.

The AI Gateway is the connective tissue across all tiers — it routes to the right model based on what's configured and available, transparently.

## Delivery: Local Add-on

The initial release ships as a Local add-on, not a core change. This is the right wedge:
- **Distribution:** Available in the Local add-on marketplace; users choose to install it. Once installed, fleet-wide services (Vector DB, MCP server) start automatically when Local launches — the same lifecycle as the Router. Per-site content indexing triggers when a site starts — at 184 docs/sec, indexing completes in seconds and is invisible to the user. Structural context (themes, plugins, config) is indexed for all sites regardless of running state. Individual services can be disabled or stopped from the Local UI.
- Follows established Local patterns; the Lightning Services architecture already supports fleet-wide services (the Router is the existence proof)
- Can iterate independently of Local's core release cycle
- Proves the pattern before any deeper integration

## The Interface Layer

The host is the product. Interfaces are surfaces into it.

**Local UI** becomes the control plane — fleet health, configuration, managing what's exposed to which AI clients, reviewing activity. Less workbench, more router admin panel.

**MCP as the universal adapter** — one great MCP surface lets Claude, Claude Code, ChatGPT, Cursor, and future clients connect without building separate integrations for each. The interface ecosystem comes to Local.

**WP Admin** — AI capabilities surface where editors and less technical users already work. Local's intelligent host layer backs WP Admin's AI through a plugin, making WordPress 7's built-in AI more capable because Local is underneath it.

**External AI clients** — developers who prefer to work in Claude Code or Cursor never need to open Local UI; they talk to the fleet through MCP.

### The Persona Split
| Persona | Primary Interface |
|---|---|
| Developer | Claude Code / Cursor via MCP |
| Freelancer / Builder | WP Admin |
| Agency / Multi-site Manager | Local UI control plane |

## Relationship to WordPress 7

WordPress 7 adds site-scoped AI — an assistant that knows one site's content, helps with block editing, content generation, SEO. Valuable but bounded by the single-site horizon.

Local's opportunity is everything WordPress 7 can't do:

- **Cross-site intelligence** — patterns, problems, and insights across your whole fleet
- **Local AI runtime** — be the on-device backend that powers WordPress 7's AI features without API calls leaving the machine
- **Context aggregation** — shared knowledge across sites; new sites don't start from zero
- **Orchestration above site-level AI** — workflows that span multiple sites; WordPress 7's per-site AI as instruments, Local as the conductor
- **Dev/test environment** for WordPress 7 AI features before they go to production

**The clean distinction:**
> WordPress 7 makes individual sites AI-native.
> Local makes the developer's entire WordPress practice AI-native.

## Strategic Notes

**The architectural foundation:** Lightning Services already manages fleet-wide services — the Router is the existence proof. Adding AI services follows an established pattern, not new infrastructure. The "effortlessly" promise is lower risk to deliver because the plumbing is already there.

**The WP Engine cloud gateway:** Tightens the Local-WP Engine relationship in a new way. Today Local and WP Engine are connected but separable. A cloud AI gateway that adds real value during local development — and works identically in production — makes the connection considerably stronger. It also creates a natural packaging opportunity: AI credits bundled with hosting plans, accessible from day one of local development.

**The MCP bet:** Building one great MCP surface means Local doesn't need N integrations for N AI clients. The interface ecosystem comes to Local. And if Local becomes the AI host that all preferred tools connect to, switching away means rewiring every integration — stronger and more natural stickiness than "easiest way to run WordPress locally."

**The local-to-production pipeline:** Local is the on-ramp to AI-native WordPress development. WP Engine is the natural place to deploy AI-native WordPress sites. The pipeline already exists — extend it to include AI infrastructure, and the same gateway, the same models, the same context follow the site from development to production.

**Resource management:** Tier 1 AI services add zero additional OS processes. The addon runs inside Electron's main process via `require()` — LanceDB, ONNX Runtime, and the embedding model are library code, not separate processes. Measured resource cost: ~90MB RSS for the embedding model, ~134MB for the vector DB with 7,500 documents across 15 sites, and ~12MB disk per 500-doc site index. A developer with 15 sites has one embedding model instance, one vector DB instance with per-site tables, and zero additional processes. Content extraction uses the site's already-running MySQL via socket connection (0.8ms per post) — no new services needed. The only component that adds a process is Tier 3 Ollama, which is inherently fleet-wide and typically already running as a user-managed system service.

**The AI Gateway is the hardest component:** The Gateway concept appears simple (route requests to the right backend) but is doing the most work in this architecture. It must abstract over fundamentally different providers (local Ollama, WP Engine cloud, OpenAI, Anthropic), handle credential lifecycle, track costs, manage fallbacks when a provider is unavailable, and present a consistent API surface. This should be sequenced last and built incrementally — start with a single provider, add routing later. The temptation to build the universal gateway first should be resisted.

**Distribution is validated — not a risk:** The addon ships ~115 MB of platform-specific assets per target (LanceDB native bindings, ONNX Runtime, quantized embedding model). This is smaller than every existing Local addon on a typical machine (median addon: 468 MB, largest: 1.27 GB). The build pipeline produces per-platform packages: LanceDB already ships per-platform via npm optional dependencies, and ONNX Runtime's all-platform 210 MB package is reduced to ~35 MB per platform by stripping non-target binaries (`rm -rf` in CI — no complex tooling). Both packages use N-API (ABI-stable), and Local's Electron ^37.8.0 is well above ONNX Runtime's recommended minimum of Electron v28+. Existing addons already ship native `.node` modules (fsevents, keytar), so there is established precedent. Zero first-run friction — no downloads, no network dependency.

**Technical feasibility is fully validated:** Five technical spikes have been executed, all producing Go results with high confidence. The complete Tier 1 stack — LanceDB vector store, ONNX Runtime CPU embeddings, MySQL content extraction, per-platform distribution — has been validated with performance margins of 10x or more on every target metric. Tier 3 Ollama integration is confirmed as a simple detect-and-connect pattern. All component licenses are permissive (Apache 2.0 or MIT). The remaining work is product and engineering: defining V1 feature scope, building the addon, validating in Electron (spikes ran in Node.js), and setting up per-platform CI/CD.

**Build principle:** One intelligent host, multiple surfaces. Every capability built into the host should be accessible from any interface. No fragmentation.
